# Hover Events in Non-Activating NSPanel WebViews

## Problem

Mono Tray is a macOS menu bar app built with Tauri. The popup window is converted from an `NSWindow` to an `NSPanel` subclass at runtime, configured as a non-activating panel (`NSNonactivatingPanelMask`). This allows the panel to appear without stealing focus from the user's current application.

However, this breaks all mouse movement event delivery to the WKWebView inside the panel:

- CSS `:hover` pseudo-class never fires
- JS `mouseover`, `mouseout`, `mouseenter`, `mouseleave` never fire
- JS `pointermove` never fires
- JS `mousemove` never fires

The root cause is that WebKit's event dispatch depends on the window being the key window or at minimum receiving `mouseMoved:` NSEvents. A non-activating panel at `NSStatusWindowLevel` does not receive these events through the normal responder chain.

## Approaches That Don't Work

1. **`setAcceptsMouseMovedEvents: true`** on the NSPanel — has no effect because the window never becomes key in the normal flow, so AppKit doesn't route mouseMoved to it.

2. **Adding `NSTrackingArea`** to the content view — the tracking area fires for the native content view but the events still don't propagate into WebKit's event handling.

3. **Overriding `sendEvent:`** to forward mouseMoved to super — the events never arrive at `sendEvent:` in the first place, so there's nothing to forward.

4. **`setBecomesKeyOnlyIfNeeded: false`** — doesn't help because the non-activating panel mask prevents the normal key-window acquisition path.

5. **JS `pointermove` / `mousemove` listeners** — these depend on WebKit receiving the native events, which it doesn't. No native event means no JS event.

## Solution: Native Global Monitor + JS Polyfill

The working solution bypasses WebKit's event delivery entirely using a two-part architecture:

### Part 1: Rust — Global NSMouseMovedMask Monitor

Register a global event monitor using `[NSEvent addGlobalMonitorForEventsMatchingMask:handler:]` for `NSMouseMovedMask` (bit 5). This receives ALL mouse movement events system-wide, regardless of which window or application is focused.

In the handler:
1. Check if the panel is visible
2. Get the cursor position via `[NSEvent mouseLocation]` (screen coordinates)
3. Get the panel's frame
4. If cursor is inside the frame, convert to window-local coordinates (flip Y for web origin) and emit a Tauri event `mono-tray://mousemove` with `{ x, y }`
5. If cursor is outside, emit `mono-tray://mouseleave`

```rust
fn install_mouse_move_monitor(window: tauri::WebviewWindow) {
    let block = block::ConcreteBlock::new(move |_event: id| {
        // ... get mouse_loc, frame, check inside ...
        if inside {
            let local_x = mouse_loc.x - frame.origin.x;
            let local_y = frame.size.height - (mouse_loc.y - frame.origin.y);
            let _ = win.emit("mono-tray://mousemove", json!({ "x": local_x, "y": local_y }));
        } else {
            let _ = win.emit("mono-tray://mouseleave", ());
        }
    });

    unsafe {
        let mask: u64 = 1 << 5; // NSMouseMovedMask
        let _: id = msg_send![ns_event_class,
            addGlobalMonitorForEventsMatchingMask:mask handler:&*block];
        std::mem::forget(block);
    }
}
```

### Part 2: JS — `.hover` Class via `elementFromPoint`

CSS `:hover` will never work, so every hover rule is duplicated with a `.hover` class:

```css
.control-btn:hover, .control-btn.hover {
  background: var(--accent-dim);
}
```

The JS side listens for the native Tauri events and uses `elementFromPoint()` to resolve which element is under the cursor:

```typescript
listen<{ x: number; y: number }>('mono-tray://mousemove', (event) => {
  const { x, y } = event.payload;
  const el = document.elementFromPoint(x, y);
  const target = el?.closest?.(hoverSelectors) ?? null;
  if (target !== currentHover) {
    currentHover?.classList.remove('hover');
    target?.classList.add('hover');
    currentHover = target;
  }
});

listen('mono-tray://mouseleave', () => {
  currentHover?.classList.remove('hover');
  currentHover = null;
});
```

## Other Required NSPanel Configuration

The panel class is registered at runtime via `objc::declare::ClassDecl` as a subclass of `NSPanel`:

- **`canBecomeKeyWindow` -> YES** — required for keyboard input (search box, etc.)
- **`canBecomeMainWindow` -> YES** — same
- **`NSNonactivatingPanelMask`** (bit 7 on styleMask) — panel doesn't activate the owning app
- **`setFloatingPanel: true`** — floats above other windows
- **`setHidesOnDeactivate: false`** — stays visible when app loses focus
- **`setWorksWhenModal: true`** — works even during modal dialogs
- **`setBecomesKeyOnlyIfNeeded: false`** — accepts key status for typing
- **Window level set LAST** — Chromium checks `window.level > NSNormalWindowLevel` to decide mouse event delivery; setting level after all other config ensures it sticks

## Key Insight

The global monitor approach works because it operates at the AppKit level, completely outside of WebKit's event dispatch pipeline. We never try to make WebKit receive mouse events — instead, we observe them independently and synthesize the hover state in JS. This is robust because `elementFromPoint()` works regardless of how the coordinates were obtained.
